## Codebase Patterns (Study These First)

- **BattlerAnim hit reactions bypass `self.play()` — reconnect `hit_received` in subclass `setup()`.** Base `BattlerAnim.setup()` wires `battler.hit_received → _anim.play("hurt")` directly on the internal AnimationPlayer, so overriding `self.play(anim_name)` in a subclass only catches manual callers, not combat-driven hits. To add procedural hit reactions (shake/flash/particle/screen-shake), override `setup()` and attach a second listener on `hit_received`. The base's 0.1s empty "hurt" shell fires harmlessly alongside — both listeners run. See `toki_battler_anim.gd::setup` + `_play_hurt_effect`. Same pattern applies to `health_depleted` (die) and `selection_toggled` (move_forward/move_to_rest) if those need procedural override.
- **`Tween.parallel().tween_property(X)` parallelizes X with the PREVIOUS tweener only.** To make a long-running property (e.g. a modulate flash) overlap several chained short-duration steps (e.g. a 6-step position shake), place the `.parallel()` call immediately after the FIRST sequential step, with the parallel step's duration set to the full sequence length. Subsequent `.tween_property(...)` calls chain sequentially behind step 1 (NOT behind the parallel one), but the parallel tweener keeps running independently until its duration expires. Example: `create_tween().tween(pos1, step).parallel().tween(modulate, 6*step).tween(pos2, step).tween(pos3, step)` → pos steps chain while modulate spans all of them.
- **gdUnit4 runs headless with `--ignoreHeadlessMode`.** The addon refuses `--headless` by default (exits 103 with an InputEvent warning). The canonical test invocation is `godot --headless --path . -s -d res://addons/gdUnit4/bin/GdUnitCmdTool.gd -a res://test --ignoreHeadlessMode`. `-a <path>` adds a test root to scan; `test/gdunit4.cfg` is a project-level manifest documenting which roots exist (gdUnit4 itself does not read it). Autoloads still initialize during the run — World/SaveSystem/TokiSave are live inside suites, so tests get a booted engine without any bootstrap code. An empty suite exits 0 with `No test cases found` — CI must treat that as green.
- **Dialog → save wiring via FieldEvents bus.** Dialog triggers (set_flag, give_item, add_party, advance_quest) are fired from `dialog_overlay.gd::_fire_triggers` as signals on the `FieldEvents` autoload. `toki_save.gd::_ready` connects to those signals and routes them to its typed mutators (`set_flag`, `give_item`, `add_to_party`, `advance_quest`). This keeps the overlay decoupled from persistence — any future system (quest tracker, achievements) can subscribe to the same signals.
- **Persistence key layout.** `TokiSave` namespaces everything under `"toki:<sub_key>"` in `SaveSystem` (flags, quests, inventory, party, current_region_id, player_tile). Dictionaries and Arrays are round-tripped whole — read via `_get_dict`/`_get_array`, written via `_set_dict`/`_set_array`.
- **Dialog selection gate currently reads `Engine.get_meta("flag_*")`, NOT TokiSave.** `region_builder.gd::_select_dialog_for` uses process-scoped Engine meta to evaluate `when_flags`. This keeps working within a single session because `dialog_overlay.gd` writes both Engine.meta AND emits FieldEvents, but on a fresh boot with persisted `starter_chosen=true` the gate would fall back to `false`. Future story should migrate the gate to `TokiSave.get_flag()`.
- **Content pipeline: JSON → tres is generated.** Authors edit `content/spine/**/*.json`; `tools/build_spine.gd` enriches with Tatoeba TP translations and emits `content/generated/**/*.tres`. Any time you touch a JSON file under `content/spine/`, rerun `godot --headless --path . --script res://tools/build_spine.gd` or the runtime still sees the stale tres. Fields added to schema resources (`schema/*.gd`) must be mirrored in `from_dict` to survive the pipeline.
- **Dialog choices schema.** DialogResource supports a `choices: Array` of `{label: {tp,en}, glyph, triggers: {set_flag|give_item|add_party|advance_quest}}`. When `choices` is non-empty, the overlay shows lettered options (A/B/C) after the last beat, picks one on keypress, and fires the picked choice's triggers IN ADDITION TO the dialog-level triggers. Dialog-level triggers still fire unconditionally — put shared side-effects there (flag flip, handing the poki) and branch-specific side-effects inside choices (which starter joins the party).
- **Combat rewards hook from the spawner side, not from combat.gd.** `Combat._on_combat_finished` frees `_battler_roster` before emitting `CombatEvents.combat_finished`, so by the time the signal reaches listeners the enemy composition is gone. The spawner (e.g. EncounterWatcher) should stash pending info (`_pending_wild_species`) at the moment it emits `combat_triggered`, then listen for `CombatEvents.combat_finished(is_player_victory)` and consume the stash. Each spawner owns its own reward hook — trigger-based combats, dialog-based combats, and wild encounters can each reward differently without entangling combat.gd.
- **`TokiSave.current_region_id` is authored by RegionBuilder on load.** Every `RegionBuilder._ready` sets `TokiSave.current_region_id = region.id` after resolving warp-meta / start-region fallback. Any system that needs "where is the player right now?" (defeat warp, save-resume, HUD) reads it from TokiSave — no need to walk the scene tree for the active Map. The region's own `spawn: Vector2i` is the canonical safe-point within that region; warp back to `World.find_region(current_region_id).spawn` to restore the player to the village center.
- **Combat reward info travels on the CombatArena, not on a signal.** `CombatArena.xp_yield: int` is set by whichever spawner builds the arena (currently `TokiArenaBuilder` lifts it from `wild_species.xp_yield`). `combat.gd::setup` snapshots it into `_pending_xp_yield` at arena-instantiate time, and the post-victory path uses it directly to drive `TokiSave.grant_xp_to_lead`. Benefit over the old "spawner listens for combat_finished" approach: combat.gd now owns the complete victory UX (XP grant → snapshot diff → VictoryPanel sequence), and any future spawner (trigger-based combat, dialog-based combat, boss fight) only has to set `arena.xp_yield` — no per-spawner reward-hook code.
- **Pathfinder collision flows through the tileset's `IsCellBlocked` custom data layer.** `GameboardLayer.is_cell_clear` reads `tile_data.get_custom_data("IsCellBlocked")` as a bool: `true` → cell is blocked (pathfinder excludes it), absent/false → cell is walkable. Every TileSet used on a Gameboard must declare `custom_data_layer_0/name = "IsCellBlocked"` with `type = 1` (TYPE_BOOL) in its .tres, and each solid atlas coord must carry `<x>:<y>/0/custom_data_0 = true`. Walkable cells omit the key. Tall-grass and other "fires an effect on step" tiles must stay walkable — the `tall_grass` tag is a separate concern from collision and lives only in TileKeys (not in the tileset custom data).
- **Rival set-piece combat rides on WarpWatcher, not FieldTriggerWatcher.** Region JSON has a `rival_fights: Array` of `{id, tile, npc_id, flag}` entries; `RegionResource.rival_fights` round-trips them via `from_dict`. WarpWatcher.initialize indexes rival cells alongside warp cells; `_on_player_arrived` checks rival cells first (so a rival can share a tile with or sit in front of a warp without leaking). On arrival, `_maybe_fire_rival` reads the NPC's first team slot (species/level/moves) from `region.npcs`, looks up the lead from `TokiSave.party()`, asks `TokiArenaBuilder.build_arena_for_rival(...)` for a PackedScene, and fires `FieldEvents.combat_triggered`. A `_pending_rival_flag` field remembers which flag to set, then `_on_combat_finished(true)` writes the flag via `TokiSave.set_flag(flag, true) + TokiSave.save()` and returns early — skipping the defeat-warp branch. On defeat, the pending flag is cleared without setting, and the normal village-respawn path runs. Reason to live in WarpWatcher rather than FieldTriggerWatcher: (1) rival fights and warps compete for the same arrival path and must resolve atomically; (2) FieldTriggerWatcher's `kind: "combat"` expects a static arena .tscn, not a dynamically-built arena from NPC team data; (3) once-per-save gating via TokiSave flags is cleaner than FieldTriggerWatcher's Engine.meta once-per-session gating.
- **TokiArenaBuilder has two static builders now.** `build_arena_for_encounter(wild_species, level, lead_species, lead_level)` — wild encounters, player gets a PokiThrow action. `build_arena_for_rival(enemy_species, level, move_ids, enemy_name, lead_species, lead_level)` — rival fights, NO PokiThrow (can't catch a rival's creature), enemy action list built from the authored NPC move_ids via `_build_actions_from_ids` instead of the species learnset. `xp_yield` is still lifted from the enemy species so combat.gd's victory panel works the same for both paths.
- **Flee (wild-only) short-circuits combat via Engine meta + combat_fled signal.** `CombatArena.allow_flee: bool` is set true by `_build_arena_node` (wild) and left false by `_build_rival_arena_node` (rival). The wild builder appends a `FleeAction` (target_scope=SELF) to the player's actions; rivals don't get one. On execute, FleeAction rolls `BASE_SUCCESS_RATE=0.75` and, on success, sets `Engine.set_meta("combat_flee_requested", true)`. `Combat._play_next_action` reads that meta BEFORE the defeat/victory checks, clears it, sets `_fled=true`, and calls `_on_combat_finished(false)`. The finished handler's `elif _fled:` branch shows a short "Got away safely!" panel and skips both `heal_party()` and the defeat dialog. Before the generic `combat_finished(false)` fires, `CombatEvents.combat_fled` emits so WarpWatcher can set its own `_fled` flag and skip the faint-warp path. Field.gd's `combat_finished` listener (show + input_paused false) still runs unchanged — a flee is a non-victory combat end from the field's perspective. Keeping flee as a "is_player_victory=false + auxiliary fled signal" preserves backward compat with every existing combat_finished listener while giving defeat-specific listeners (WarpWatcher) a discriminator.
- **TileKeys is biome-aware.** `tools/tile_keys.gd` now exposes `resolve(key, biome="town")` with a matching `FOREST_KEYS` table. Town is the default so existing callers are unchanged; forest regions can call `TileKeys.resolve(k, "forest")` to remap shared keys (`grass`, `tree`, `tall_grass`, `path_*`, `water`) onto the `content/tilesets/forest_summer.tres` source ids. When adding a new biome (water, ice, cave), add a SOURCE_<BIOME>_* constants block, a `<BIOME>_KEYS` dict, and extend `_lookup` — no changes needed in `region_builder.gd` or `encounter_watcher.gd`.
- **SaveSystem addon requires a pre-seeded namespace OR writes silently no-op.** The addon's `set_var(key_path, value)` has a guard at line 175 (`if not has(key_path): return`) that means hierarchical writes (e.g. `toki:current_region_id`) only succeed when the key's parent dict already contains that key. `TokiSave._ready()` therefore calls `_ensure_namespace()` to seed `SaveSystem.current_state_dictionary["toki"]` with empty shells for flags / quests / inventory / party / current_region_id / player_tile. This bootstrap runs after SaveSystem's own `_ready` (autoload order in project.godot: SaveSystem at line 34, TokiSave at line 36), so any values loaded from `user://save_data.sav` are preserved — `_ensure_namespace` only populates missing keys. Without this seed, every autosave call writes `{}` to disk and Continue has nothing to restore. Also note: SaveSystem.`default_file_path` is `"user://save_data.sav"` — `TokiSave.SAVE_PATH` MUST match this constant because SaveSystem's private `_load()` runs against the default path in its `_ready()`. A mismatch (e.g. the earlier `"user://save.tres"`) means writes go to one file, auto-loads come from another — nothing round-trips across a reboot.
- **`TokiSave.player_tile` is synced centrally from the TokiSave autoload, not from region watchers.** `TokiSave._ready` subscribes to `Player.gamepiece_changed` and then follows the active gamepiece's `arrived` signal, writing `player_tile = Gameboard.get_cell_under_node(gp)` on every step. Region-scoped listeners (EncounterWatcher, WarpWatcher, FieldTriggerWatcher) die on scene reload — TokiSave's autoload scope survives, so the sync is uninterrupted across warps. Disk flush still piggybacks on region-crossing / combat-end autosaves (US-031/US-032); we intentionally don't save_data.sav every tile. If a future story needs per-tile durability, debounce inside TokiSave's `_on_player_arrived`, don't fan the save call out to watchers.
- **Continue routes through the warp_target Engine.meta channel.** The title scene's Continue button sets `Engine.set_meta("warp_target_region", TokiSave.current_region_id)` and `"warp_target_tile"` (from `TokiSave.player_tile` when non-zero) before `change_scene_to_file(GAME_SCENE)`. RegionBuilder._ready's existing warp-target consumption path handles both — it already falls back to `World.start_region_id` and `region.spawn` respectively, so Continue on an old save (pre-US-039, no player_tile written) lands at region.spawn without special casing. The dialog selection in `_select_dialog_for` now reads `when_flags` from `TokiSave.get_flag()` instead of `Engine.get_meta("flag_*")`, so persisted flags (starter_chosen etc.) survive a Continue boot and prevent re-firing of one-shot dialogs.
- **Hit effectiveness flows via BattlerHit → `hit_landed` signal → UIDamageLabel.** `AttackBattlerAction.execute` computes `Elements.multiplier(element, target.stats.affinity)`, multiplies damage by it, and passes the multiplier into `BattlerHit.new(dmg, to_hit, effect)`. `Battler.take_hit` emits `hit_landed(damage, effectiveness)` alongside the existing `hit_received(damage)` — both signals fire on every successful hit. The legacy `hit_received` stays because `battler_anim.gd` and `ui_life_bar.gd` connect to it with `(value: int)` signatures; adding a second positional param would break those lambdas at runtime. `UIEffectLabelBuilder` swapped onto `hit_landed` and forwards effectiveness into `UIDamageLabel.setup(origin, amount, effectiveness)`, which picks between damage/heal/super-effective (emerald)/resisted (grey) colors. Note: Godot's theme color override key is `font_outline_color` (American) — the older codebase had `font_outline_colour` (British), which silently no-ops.
- **Two-channel battler sprites.** `SpeciesResource` exposes BOTH `sprite_frame: int` (Kenney dungeon-atlas cell, used by NPCs in `region_builder.gd`) and `sprite_src: String` (res:// path to a creature sprite sheet in `assets/creatures/`, used by combat battlers). `TokiBattlerAnim` has matching methods `set_sprite_frame(frame)` and `set_sprite_src(path)`; the latter loads the sheet and shows its first 64×64 cell at 2× scale. `TokiArenaBuilder._configure_anim` prefers `sprite_src` when non-empty and falls back to the atlas frame, so species authored without `sprite_src` still render. The combat pipeline and the overworld NPC pipeline stay independent — authoring a new species with a unique battle sprite only requires dropping a PNG into `assets/creatures/<category>/` and adding one `"sprite_src"` line to the species JSON.
- **Biome-driven tileset selection in RegionBuilder.** `RegionResource.biome: String` ("town"/"forest"/…) drives two things in `_build_tilemap` → `_resolve_tileset()`: (1) the TileSet to paint the ground with, via `BIOME_TILESETS` const (town → `overworld/maps/tilesets/kenney_terrain.tres`, forest → `content/tilesets/forest_summer.tres`); and (2) the biome arg passed to `TileKeys.resolve(key, region.biome)` inside `_paint_layer`, which remaps shared keys (grass/tree/tall_grass/path/water) onto the chosen atlas' source ids. The scene-assigned `@export var tileset` is the fallback — kept so existing scenes (`toki_town_main.tscn`) work unmodified. Authoring flow to add a new region biome: add entries to a `TileKeys.FOREST_KEYS`-style table, add `SOURCE_<BIOME>_*` constants, extend `TileKeys._lookup`, and add one line to `BIOME_TILESETS`. No per-region .tscn work required. `RegionResource.music_track: String` is the companion new export and is authored alongside biome — it's read by a future music system; `""` means "leave current track playing."

## 2026-04-19 - US-019
- Implemented battler hit reaction: 0.15s horizontal shake (damped oscillation: +6, -6, +3.6, -3.6, +1.8, 0 px) with a parallel white modulate flash (Color(1.8, 1.8, 1.8) → Color.WHITE) on the Sprite2D child.
- Files changed:
  - `src/combat/battlers/toki_battler_anim.gd` — overrode `setup(battler, facing)` to connect `hit_received` → `_play_hurt_effect()`; overrode `play(anim_name)` to intercept `"hurt"` for the same effect; added `_play_hurt_effect()` which kills any in-flight hurt tween, resets sprite position + modulate, and runs a 6-step chained damped position oscillation in parallel with a single full-duration modulate-to-white tween (parallel group placed after the first position step so the flash overlaps every subsequent position step).
- **Learnings:**
  - **Base `BattlerAnim.setup()` wires `battler.hit_received → _anim.play("hurt")` directly on the internal AnimationPlayer**, bypassing any override of `self.play()` in a subclass. Two ways to react to hits from a subclass: (a) connect `hit_received` again in the subclass's `setup()` override (what US-019 did — a second listener runs alongside the base one, which is fine because the base `"hurt"` animation is a 0.1s empty shell), or (b) override `play()` for manual callers. Having both means any caller path lights the effect.
  - Over-unit `Color(1.8, 1.8, 1.8)` on `CanvasItem.modulate` clamps by driver but produces a visible white-out flash against the base-colored sprite — Godot does NOT clamp the value at assignment time, so tweening back to `Color.WHITE` (1,1,1) is a smooth transition. If HDR goes off or a different render backend caps, the effect still reads as "brighten then normal" because values >1 still exceed the texture color.
  - **`Tween.parallel().tween_property(X)` parallelizes X with the PREVIOUS tweener, not with the whole tween.** For a modulate flash that should span multiple chained position steps, call `.parallel().tween_property(modulate, FULL_DURATION)` immediately after the FIRST `.tween_property(position, step)`. Subsequent `.tween_property(position, step)` calls chain sequentially, but because the modulate's `FULL_DURATION` is 6× one position step, the modulate tween keeps running in parallel with position steps 2–6. Pattern in code: `create_tween()` → `tween(pos1, step)` → `.parallel().tween(modulate, FULL)` → `tween(pos2, step)` → `tween(pos3, step)`… — the sequential chain resumes behind pos1, not behind the modulate. The flash now reads as "brighten across the full shake" rather than "flash only on settle."
